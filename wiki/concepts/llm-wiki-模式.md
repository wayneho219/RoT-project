# LLM Wiki 模式

## 一句話定義

讓 LLM 擔任「編譯器」，把原始資料一次性整理成結構化、互相連結的 wiki，之後的查詢從 wiki 讀取而非原始資料，使知識隨時間累積。

## 與 RAG 的對比

| 面向 | 傳統 RAG | LLM Wiki |
|------|----------|----------|
| 知識處理時機 | 每次查詢時即時推導 | 收集時預先編譯 |
| 知識積累 | 不積累，每次從零 | 持續累積，越用越豐富 |
| 查詢速度 | 需要向量搜尋 + 重新合成 | 讀結構化頁面，已預先整理 |
| 矛盾偵測 | 不主動偵測 | Lint 階段主動標記 |
| 個人規模需求 | 需要向量資料庫、embedding | 只需要 markdown 目錄 |

## 三個層次

1. **原始資料（raw/）**：使用者收集的資料，LLM 只讀不改
2. **Wiki**：LLM 生成的 markdown 頁面，LLM 完全擁有這層
3. **Schema（SCHEMA.md）**：規範 wiki 的慣例，使用者與 LLM 共同演化

## 三個操作

- **Ingest（收集）**：新資料進來 → 寫摘要頁 → 更新相關頁面 → 更新索引
- **Query（查詢）**：讀 index.md → 找相關頁 → 合成答案 → 可存回 wiki
- **Lint（維護）**：掃描矛盾、孤立頁面、缺漏概念、過時資訊

## 為什麼能持續維護

傳統人工 wiki 的致命弱點是維護成本：人會忘記更新交叉引用、覺得書記工作無聊而放棄。LLM 不會厭倦，不會忘記更新引用，因此 wiki 能真正持續成長。

## 知識累積的飛輪

```
新資料 → 編譯進 wiki
查詢答案 → 存回 wiki
Lint 發現連結 → 寫新概念頁
                ↓
          wiki 越來越豐富
                ↓
          未來查詢越來越準確
```

## 規模限制與應對

- 個人規模（<1000 頁）：純 markdown 即可，不需向量資料庫
- 頁面超過 ~150 頁：index.md 分片（shard）
- 頁面超過 ~300 頁：啟用 `wiki_search.py` BM25 搜尋
- 每頁軟上限 400 行，硬上限 800 行（超過需拆頁）

## 失敗模式

- **靜默污染**：誤讀一個資料來源 → 錯誤進入 wiki → 影響後續編譯。對策：每個聲明都標注來源（`sources:` frontmatter）
- **自我引用漂移**：LLM 開始把 wiki 當成真理而非參考。對策：更新頁面時重新核對原始資料
- **維護棘輪**：wiki 越大、lint 越多、人工審查負擔越重。對策：規律 lint、及時分頁
